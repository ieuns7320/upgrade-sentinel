# UpgradeSentinel 프로젝트 1차 진행 회고

**Created_Date**: 2026_04_28

## 1. 프로젝트 시작 배경

이번 프로젝트는 Rust/Go 일부 혼합 기반 업그레이드형 스마트컨트랙트 보안 분석기를 목표로 시작했다.
단순히 Web3 서비스를 만드는 것이 아니라, 업그레이드형 컨트랙트에서 발생할 수 있는 보안 이상 징후를 정적 분석과 온체인 정보 수집을 통해 탐지하는 도구를 만드는 것을 목표로 잡았다.

초기 방향은 다음과 같이 정리했다.

* Rust: 정적 분석 코어 담당
* Go: RPC 기반 온체인 정보 수집 담당
* 분석 대상: 업그레이드형 스마트컨트랙트
* 1차 규칙 후보:
    * Initializer 노출 여부
    * Implementation 직접 초기화 가능성
    * Upgrade 경로 접근제어 이상
    * Delegatecall / low-level call 위험
    * Selfdestruct 경로
    * EIP-1967 slot 확인

처음부터 모든 기능을 구현하려고 하지 않고, 먼저 프로젝트 구조와 최소 실행 가능한 분석 파이프라인을 만드는 것을 첫 목표로 설정했다.

---

## 2. 이번 단계에서 진행한 작업


### 2-1. 프로젝트 구조 초기화

가장 먼저 루트 프로젝트 디렉터리를 생성하고, 아래와 같은 기본 구조를 만들었다.

* rust-analyzer/
* go-collector/
* samples/
* docs/
* schema/
* README.md
* docs/spec.md

이 과정에서 프로젝트의 전체 역할 분리를 먼저 고정했다.
이렇게 구조를 먼저 만들고 시작하니, 이후 어떤 파일을 어디에 둘지 훨씬 명확해졌다.

---

### 2-2. README 및 spec 문서 작성

구현에 들어가기 전에 프로젝트의 목적과 범위를 README와 spec 문서에 먼저 정리했다.

README에는 다음 내용을 담았다.

* 프로젝트 한 줄 설명
* Rust/Go 역할 분리
* 탐지하려는 초기 위험 패턴
* 디렉터리 구조
* 현재 상태(bootstrap phase)

docs/spec.md에는 아래 내용을 정리했다.

* 프로젝트 목표
* Rust analyzer와 Go collector의 역할
* 1차 탐지 규칙 목록
* 출력 형식(CLI / JSON / Markdown)

이 과정을 통해, 단순히 “코드만 짜는 프로젝트”가 아니라 무엇을 만들고 있는지 문서로 설명 가능한 상태를 먼저 만들 수 있었다.

---

### 2-3. Git 저장소 초기화 및 원격 저장소 연결

프로젝트 루트에서 Git 저장소를 초기화하고 main 브랜치로 설정했다.
이후 GitHub 원격 저장소를 연결하여 push까지 진행했다.

이 과정에서 단순히 커밋만 하는 것이 아니라, 프로젝트 루트 전체가 하나의 저장소로 관리되도록 정리하는 과정도 함께 진행했다.

---

### 2-4. Rust analyzer 최소 실행 뼈대 구현

rust-analyzer에서는 우선 다음 수준까지 구현했다.

* Cargo 프로젝트 생성
* CLI 인자 처리
* AST JSON 파일 경로 입력받기
* JSON 파일 읽기
* serde, serde_json를 사용해 구조체로 deserialize
* AST 내 ContractDefinition / FunctionDefinition 추출
* contract / function 이름 출력

초기에는 단순 JSON 로딩만 목표로 했지만, 이후 AST 최소 모델을 구성하면서 contract와 function 목록을 순회할 수 있는 수준까지 확장했다.

즉, 이 단계에서 **“AST를 입력으로 받아 정적 분석용 데이터 구조로 해석하는 최소 뼈대”**를 만들었다.

---

### 2-5. 첫 번째 룰(UFG-001 / Initializer Exposure 후보) 구현

그 다음으로, 첫 번째 보안 탐지 규칙인 UPG-001의 가장 단순한 1차 버전을 구현했다.

현재 구현한 기준은 다음과 같다.

* 함수 이름이 initialize 또는 init인 경우
* initializer-like function 후보로 간주
* finding 생성
* CLI 출력
* JSON 직렬화 출력

이 단계에서 finding.rs와 rules.rs를 분리하여,

* AST 로드
* 룰 실행
* finding 생성
* JSON 출력

까지 하나의 파이프라인으로 연결했다.

아직은 이름 패턴 기반의 단순 탐지지만, 이 작업을 통해 분석기가 실제로 “탐지 결과”를 출력하는 형태를 처음으로 갖추게 되었다.

## 3. 진행 중 발생한 문제와 해결 과정

### 3-1. 원격 GitHub 저장소 push 거절 문제

원격 저장소를 연결한 뒤 git push -u origin main을 수행했을 때 push가 거절되었다.

메시지는 대략 다음 의미였다.

* 원격 저장소에 이미 로컬에 없는 작업이 존재함
* fetch/pull 먼저 하거나, 강제로 push해야 함

원인은 GitHub에서 repository를 만들 때 README 등 초기 파일이 자동 생성되어, 원격과 로컬의 히스토리가 서로 다른 상태였기 때문이었다.

프로젝트 초기 단계였고, 원격의 내용이 중요하지 않았기 때문에 --force-with-lease를 사용해 로컬 상태를 기준으로 push했다.

이를 통해 로컬 프로젝트 구조를 GitHub에 정상 반영할 수 있었다.

---

### 3-4. Rust Deserialize 관련 에디터 오류

Rust 코드 작성 중 Neovim에서 #[derive(Deserialize)] 부분에 오류가 표시되었다.

에디터에 나타난 핵심 에러는 다음과 같았다.

* proc macro 'Deserialize' not expanded
* proc-macro server's api version is newer than rust-analyzer's

처음에는 코드 자체가 잘못된 것처럼 보였지만, 실제 원인은 Rust 코드 문제라기보다 에디터의 rust-analyzer 버전 충돌이었다.

즉,

* 시스템의 Rust toolchain은 더 최신
* Neovim이 사용하는 rust-analyzer는 더 구버전

인 상태라, proc macro를 올바르게 확장하지 못하고 있었던 것이다.

해결 과정은 다음과 같았다.

* cargo check로 실제 코드 컴파일 상태를 먼저 확인
* 코드 문제가 아니라 LSP 문제임을 분리해서 판단
* rust-analyzer 버전/경로 확인
* Neovim에서 사용하는 rust-analyzer를 업데이트 또는 재설치
* 이후 에디터 오류 해소

이 경험을 통해, 에디터 경고와 실제 빌드 오류를 분리해서 판단하는 습관이 중요하다는 점을 배웠다.

## 5. 이번 단계에서 배운 점

### 5-1. 처음부터 기능보다 구조를 먼저 잡는 것이 중요하다

처음에는 바로 보안 룰 구현으로 들어가고 싶었지만, 실제로는

* 디렉터리 구조
* 문서화
* Git 정리
* 입력/출력 형식 고정

이 먼저 잡혀 있어야 이후 구현이 훨씬 수월했다.

---

### 5-2. Git 문제는 대부분 구조 문제다

이번 진행에서 Git 관련 문제는 단순 명령어 실수라기보다,
저장소 경계가 어디인지 명확하지 않았던 것에서 시작됐다.

특히 하위 디렉터리에 별도 .git가 생기면 생각보다 쉽게 꼬일 수 있다는 점을 체감했다.

---

### 5-3. 에디터 에러와 실제 코드 에러를 분리해야 한다

Rust Deserialize 오류는 처음 보면 코드가 틀린 것처럼 보였지만, 실제로는 LSP 버전 문제였다.
이 경험 덕분에 앞으로는

* cargo check
* cargo run
* LSP 경고

를 분리해서 판단해야 한다는 기준이 더 명확해졌다.

---

### 5-4. 작은 규칙 하나라도 end-to-end로 연결하는 것이 중요하다

이번에 initialize 이름만 보고 finding을 만드는 규칙은 매우 단순하다.
하지만 그 단순한 규칙 하나를 통해

* AST 입력
* 모델 매핑
* 룰 실행
* finding 생성
* 결과 직렬화

까지 한 번에 연결해볼 수 있었다.

이건 이후 규칙 확장 시 그대로 재사용할 수 있는 중요한 뼈대가 된다.

---

## 6. 아쉬운 점

현재 구현된 UPG-001은 아직 매우 단순하다.

예를 들어 아직 보지 못하는 것들이 있다.

* visibility
* modifier 정보
* initializer / reinitializer 여부
* _disableInitializers() 호출 흔적
* constructor와의 관계
* 실제 외부 노출 여부

즉, 현재는 “후보 탐지기” 수준이지, 실제로 의미 있는 보안 판정기라고 보기는 어렵다.

또한 Go collector는 아직 구조만 존재하고, 실제 RPC 기반 온체인 수집 기능은 붙지 않은 상태다.

---

## 7. 다음 단계 계획

다음 단계에서는 다음 작업을 진행할 예정이다.

Rust analyzer 고도화

* FunctionDefinition에서 visibility, modifiers 추출
* UPG-001 evidence 강화
* initializer modifier 유무 반영
* 위험도 판정 정교화

이후 확장

* _disableInitializers() 흔적 탐지 시도
* upgrade 함수 접근제어 규칙 추가
* Go collector에서 EIP-1967 slot 조회 구현

즉, 다음부터는 단순 구조 구축 단계에서 벗어나 “실제 보안 규칙의 정밀도”를 높이는 단계로 넘어갈 계획이다.

---

## 8. 총평

이번 단계는 겉보기에는 단순한 bootstrap 작업처럼 보이지만, 실제로는 프로젝트 전체의 기반을 세우는 중요한 시간이었다.
Git 구조 문제, 원격 저장소 충돌, Rust LSP 버전 문제 같은 사소하지만 실제 개발 흐름을 끊는 문제들을 직접 겪고 해결하면서, 단순히 코드를 작성하는 것보다 개발 환경과 구조를 안정화하는 것이 먼저라는 점을 분명히 체감했다.

결과적으로 이번 단계의 가장 큰 성과는,
단순한 아이디어를 넘어서 실제로 AST를 받아 finding을 출력하는 업그레이드형 컨트랙트 분석기의 최소 형태를 만든 것이라고 정리할 수 있다.
