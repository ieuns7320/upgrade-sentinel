# UpgradeSentinel 프로젝트 2차 진행 회고

**Created_Date**: 2026_04_30

## 1. 이번 단계의 목표

1차 회고(001.md) 이후부터는 단순히 AST를 읽고 `initialize` 이름을 찾는 수준을 넘어서,  업그레이드형 스마트컨트랙트 보안 분석기다운 규칙을 실제로 확장하는 것을 목표로 잡았다.

이번 단계의 핵심 목표는 다음과 같았다.

* UPG-001 정교화
* UPG-002 구현
* UPG-003 구현
* UPG-004 구현
* UPG-005 구현
* Go collector를 이용한 UPG-006 1차 구현

즉, 1차 회고 시점까지는 “최소 분석기”를 만드는 단계였다면,  이번 단계부터는 **업그레이드형 컨트랙트에 특화된 보안 규칙을 실제로 하나씩 구현해 나가는 단계**였다.

---

## 2. 이번 단계에서 진행한 작업

### 2-1. UPG-001 정교화

1차 회고 시점의 UPG-001은 함수 이름이 `initialize` 또는 `init`인 경우를 단순히 후보로 잡는 수준이었다.  이번 단계에서는 이를 조금 더 현실적인 규칙으로 보강했다.

추가로 반영한 정보는 다음과 같다.

* `visibility`
* modifier 이름
* `initializer` / `reinitializer` modifier 존재 여부
* modifier 존재 여부에 따른 evidence 강화
* 상황에 따른 severity 조정

예를 들어,

* `public` / `external` 이면서 `initializer` modifier가 없으면 상대적으로 더 위험하게 보고
* `initializer` modifier가 존재하면 초기화 의도가 드러난 것으로 보고 severity를 낮추는 방식으로 정리

이 작업을 통해 UPG-001은 단순한 이름 매칭 규칙에서 벗어나,  **초기화 함수가 어느 정도 보호되고 있는지까지 함께 보는 규칙**으로 발전했다.

---

### 2-2. UPG-002 구현 – Implementation lock 부재 후보 탐지

그 다음으로는 implementation contract가 잠겨 있지 않을 가능성을 탐지하는 UPG-002를 구현했다.

이 규칙에서 확인한 정보는 다음과 같다.

* initializer-like 함수 존재 여부
* constructor 존재 여부
* constructor 내부 `_disableInitializers()` 호출 흔적

구현 흐름은 다음과 같았다.

* 계약 내에 `initialize` / `init` 계열 함수가 존재하는지 확인
* constructor가 존재하는지 확인
* constructor body 내부에서 `_disableInitializers()` 호출 흔적을 탐지
* 초기화 함수는 있는데 `_disableInitializers()` 흔적이 없으면 implementation lock 부재 후보로 finding 생성

이 규칙을 통해, 업그레이드형 컨트랙트에서 자주 문제되는  **“implementation contract가 직접 초기화 가능한 상태로 남아 있는지”**를 탐지할 수 있는 기반을 만들었다.

---

### 2-3. UPG-003 구현 – Upgrade 경로 접근제어 이상 후보 탐지

이번 단계에서 특히 업그레이드형 컨트랙트 분석기다운 느낌을 크게 만든 규칙은 UPG-003이었다.

UPG-003은 다음과 같은 upgrade 관련 민감 함수 후보를 탐지한다.

* `upgrade`
* `upgradeTo`
* `upgradeToAndCall`
* `setImplementation`

이후 다음 조건을 함께 확인했다.

* 함수가 `public` 또는 `external`인지
* modifier가 존재하는지
* `onlyOwner`, `onlyAdmin`, `adminOnly`, `onlyRole`, `auth` 등 접근제어를 시사하는 modifier가 있는지

그리고 외부에서 호출 가능한 upgrade-like 함수인데 접근제어 modifier가 보이지 않으면  **“Upgrade-like function may lack access control”** finding을 생성하도록 구현했다.

이 규칙을 통해, 분석기는 이제 단순히 초기화 관련 규칙만 보는 것이 아니라  **업그레이드 권한 경로 자체를 검토하는 단계**로 넘어가게 되었다.

---

### 2-4. UPG-004 구현 – low-level call 후보 탐지

그 다음으로는 low-level call 흔적을 탐지하는 UPG-004를 구현했다.

이 규칙을 위해 AST 모델을 확장하면서 `MemberAccess` 구조를 추가했고,  함수 body 내부에서 다음과 같은 멤버 호출을 탐지하도록 만들었다.

* `delegatecall`
* `call`

탐지 결과는 다음과 같이 정리했다.

* `delegatecall` → `HIGH`
* `call` → `MEDIUM`

이 규칙은 아직 “취약점 확정”이 아니라 **위험 후보 지점 표시** 수준이지만,  적어도 함수 body 내부 구현부까지 보기 시작했다는 점에서 의미가 있었다.

이 시점부터 분석기는 선언부 기반 규칙만 보는 것이 아니라,  **구현부 내부 호출 흔적까지 읽는 분석기**로 발전하기 시작했다.

---

### 2-5. UPG-005 구현 – destructive path 후보 탐지

UPG-005는 계약 파괴와 관련된 흔적을 탐지하는 규칙이다.

이번 단계에서는 함수 body 내부에서 다음 호출 흔적을 확인했다.

* `selfdestruct`
* `suicide`

그리고 해당 흔적이 보이면 함수 단위 finding을 생성하도록 구현했다.

이 규칙은 구현 자체는 비교적 단순했지만,  업그레이드형 컨트랙트에서 implementation 또는 민감 경로에 파괴 함수가 존재하는 경우는 매우 민감하기 때문에 의미가 컸다.

이제 분석기는 다음 영역들을 모두 다루게 되었다.

* 초기화 함수 보호
* implementation lock 여부
* upgrade 권한 경로
* low-level call
* destructive path

즉, 정적 분석기의 핵심 보안 규칙 세트가 거의 완성되기 시작했다.

---

### 2-6. Go collector 구현 – UPG-006 1차

정적 분석 규칙 확장과 별도로,  처음에 설계했던 Go 기반 온체인 수집기 또한 실제로 동작 가능한 수준까지 구현했다.

이번 단계에서 구현한 기능은 다음과 같다.

* RPC URL 입력
* 대상 주소 입력
* `eth_getCode`를 통한 코드 존재 여부 확인
* EIP-1967 implementation slot 조회
* EIP-1967 admin slot 조회
* raw slot value 출력
* slot value를 address로 해석
* JSON 형태 결과 출력

이를 통해 특정 주소가

* 실제로 컨트랙트인지
* EIP-1967 implementation slot을 사용하는지
* admin slot이 비어 있는지

같은 정보를 실제 온체인 상태를 기준으로 확인할 수 있게 되었다.

이 작업은 단순히 Go 코드를 하나 추가한 것이 아니라,  처음 프로젝트를 설계할 때 잡았던 **“Rust 정적 분석 + Go 온체인 수집” 구조가 실제로 살아난 순간**이었다.

## 3. 진행 중 발생한 문제와 해결 과정

### 3-1. Go dependency 미설치 문제

Go collector 구현 후 실행하려고 했을 때 다음과 같은 에러가 발생했다.

* `no required module provides package github.com/ethereum/go-ethereum/common`
* `no required module provides package github.com/ethereum/go-ethereum/ethclient`

처음에는 import 경로나 코드 문제처럼 보일 수 있었지만,  실제 원인은 단순히 `go-ethereum` 의존성이 아직 `go.mod`에 추가되지 않았기 때문이었다.

해결 과정은 다음과 같았다.

* `go get github.com/ethereum/go-ethereum@latest`
* `go mod tidy`

이후 다시 실행하자 정상적으로 빌드 및 실행이 가능해졌다.

이 문제를 통해, Go 프로젝트에서는 코드를 작성하는 것만으로 끝나는 것이 아니라  **필요한 모듈이 `go.mod` 기준으로 정확히 관리되어야 한다**는 점을 다시 확인했다.

## 5. 이번 단계에서 배운 점

### 5-1. 규칙 하나를 완성하는 데 필요한 정보는 생각보다 많다

처음에는 `initialize`, `upgradeTo` 같은 이름만 보면 어느 정도 탐지가 가능할 것처럼 보였다.  
하지만 실제로는 의미 있는 결과를 만들기 위해 최소한 다음 정보가 필요했다.

* visibility
* modifier
* constructor 존재 여부
* 함수 body 내부 호출 흔적
* 온체인 slot 정보

즉, 단순 이름 매칭만으로는 한계가 뚜렷했고,  조금이라도 더 현실적인 규칙을 만들려면 **문맥 정보가 반드시 필요하다**는 점을 체감했다.

---

### 5-2. 업그레이드형 컨트랙트 분석은 함수 단위만으로 끝나지 않는다

이번에 구현한 규칙들을 보면 층위가 서로 다르다.

* UPG-001: 초기화 함수 단위
* UPG-002: implementation contract 단위
* UPG-003: upgrade 권한 경로 단위
* UPG-004/005: 함수 body 내부 구현 단위
* UPG-006: 온체인 배포 상태 단위

즉 업그레이드형 컨트랙트 분석은 단순히 함수 하나만 보는 문제가 아니라,  **계약 구조, 구현부, 배포 상태를 함께 봐야 하는 작업**이라는 점이 더 선명해졌다.

---

### 5-3. 정적 분석과 온체인 정보는 서로 보완 관계다

Rust analyzer만으로도 많은 정보를 읽을 수 있었지만,  proxy 구조나 implementation slot 같은 것은 결국 온체인 상태를 봐야 더 정확해진다.

반대로 Go collector만으로는 slot 값을 알 수 있어도,  함수 보호 상태나 내부 low-level call 흔적까지는 알기 어렵다.

즉 두 축은 경쟁 관계가 아니라,  **서로 부족한 부분을 채워주는 보완 관계**라는 점이 이번 단계에서 분명해졌다.

---

### 5-4. 작은 규칙들을 쌓아야 비로소 “분석기”가 된다

UPG-001만 있을 때는 아직 “이름 매칭 도구” 느낌이 강했다.  
하지만 UPG-002, UPG-003, UPG-004, UPG-005, UPG-006이 하나씩 추가되면서  비로소 “업그레이드형 스마트컨트랙트 보안 분석기”라는 프로젝트 정체성이 살아나기 시작했다.

즉 분석기는 처음부터 완성형으로 나오는 것이 아니라,  **작은 탐지 규칙들을 일관된 구조 위에 계속 축적하면서 만들어지는 것**이라는 점을 체감했다.

---

## 6. 아쉬운 점

이번 단계에서 규칙은 많이 늘어났지만, 아직 명확한 한계도 남아 있다.

예를 들어,

* custom modifier 이름은 모두 알 수 없음
* body 내부 깊은 중첩 표현식은 아직 충분히 순회하지 않음
* `require(msg.sender == owner)` 같은 인라인 접근제어는 아직 분석하지 않음
* `_disableInitializers()` 탐지는 단순 흔적 수준에 가까움
* `delegatecall` / `call`도 target 신뢰성, return value 처리 여부까지는 아직 보지 않음
* Go collector는 beacon slot, implementation 코드 조회, proxy 유형 heuristic이 아직 없음

즉 지금은 “의미 있는 후보 탐지기”로는 발전했지만,  아직 “정밀한 판정기”라고 보기는 어렵다.

## 7. 다음 단계 계획

다음 단계에서는 다음 방향으로 확장할 계획이다.

### Go collector 고도화

* beacon slot 추가
* implementation 주소의 코드 존재 여부 확인
* proxy-like summary 추가
* EIP-1967 계열 여부를 조금 더 구조적으로 요약

### 결과 통합

* Rust analyzer 결과와 Go collector 결과를 병합할 수 있는 공통 포맷 정리
* 최종 JSON / Markdown 리포트 구조 고민

즉 다음부터는 규칙을 더 많이 늘리기보다는,  **이미 구현한 규칙과 온체인 수집 결과를 더 잘 정리하고 통합하는 방향**으로 넘어갈 계획이다.

---

## 8. 총평

이번 단계는 프로젝트가 실제 분석기다운 형태를 갖추기 시작한 시점이었다.  
초기화 함수 탐지 하나만 있던 상태에서 벗어나, implementation lock, upgrade 접근제어, low-level call, destructive path, EIP-1967 slot inspection까지 실제로 구현하면서  프로젝트의 핵심 주제가 훨씬 선명해졌다.

특히 이번 단계에서 가장 의미 있었던 점은,  처음 설계했던 **Rust 정적 분석 + Go 온체인 수집** 구조가 단순 아이디어 수준이 아니라 실제 동작하는 형태로 연결되기 시작했다는 점이다.

결과적으로 이번 단계의 가장 큰 성과는,  UpgradeSentinel이 더 이상 “AST를 읽는 실험 코드”가 아니라  **업그레이드형 스마트컨트랙트 보안 이상 징후를 여러 축에서 탐지할 수 있는 프로토타입 분석기**로 발전하기 시작했다는 데 있다고 정리할 수 있다.
